# Layout Run And Projection Model

> Status: Active (Prompts 39-44)
> Scope: Canonical layout run lineage, page-result persistence, recall/rescue contracts, activation prerequisites, and explicit document projection state

This document defines the persisted model used by the Phase 3 layout-analysis route family introduced in Prompt 39.
It now includes Prompt 44 recall-first execution requirements used before any downstream activation.

## Canonical Tables

## `layout_runs`

Document-scoped append-only layout-analysis attempts.

- `id` (PK)
- `project_id`
- `document_id`
- `input_preprocess_run_id`
- `run_kind` (`AUTO` in Prompt 39)
- `parent_run_id` (nullable)
- `attempt_number`
- `superseded_by_run_id` (nullable forward lineage link)
- `model_id` (nullable)
- `profile_id` (nullable)
- `params_json`
- `params_hash`
- `pipeline_version`
- `container_digest`
- `status` (`QUEUED | RUNNING | SUCCEEDED | FAILED | CANCELED`)
- `created_by`
- `created_at`
- `started_at` (nullable)
- `finished_at` (nullable)
- `failure_reason` (nullable)

## `page_layout_results`

Per-page status and placeholder output references scoped to one layout run.

- `run_id`
- `page_id`
- `page_index`
- `status` (`QUEUED | RUNNING | SUCCEEDED | FAILED | CANCELED`)
- `page_recall_status` (`COMPLETE | NEEDS_RESCUE | NEEDS_MANUAL_REVIEW`)
- `active_layout_version_id` (nullable)
- `page_xml_key` (nullable)
- `overlay_json_key` (nullable)
- `page_xml_sha256` (nullable)
- `overlay_json_sha256` (nullable)
- `metrics_json`
- `warnings_json`
- `failure_reason` (nullable)
- `created_at`
- `updated_at`

Prompt 39 explicitly allows these canonical output pointers to remain null when inference/output materialization is not yet implemented.

## `layout_recall_checks`

Canonical persisted recall-check results per run/page pair.

- `run_id`
- `page_id`
- `recall_check_version`
- `missed_text_risk_score`
- `signals_json`
- `created_at`

Prompt 44 requires one persisted recall-check record for every page result before activation is allowed.

## `layout_rescue_candidates`

Canonical rescue-candidate rows generated by the recall check.

- `id`
- `run_id`
- `page_id`
- `candidate_kind` (`LINE_EXPANSION | PAGE_WINDOW`)
- `geometry_json`
- `confidence`
- `source_signal`
- `status` (`PENDING | ACCEPTED | REJECTED | RESOLVED`)
- `created_at`
- `updated_at`

Prompt 44 requirements:

- candidate status starts explicit (`PENDING | ACCEPTED | REJECTED | RESOLVED`)
- `candidate_kind` is typed (`LINE_EXPANSION | PAGE_WINDOW`)
- geometry is explicit and browser-safe (`geometry_json`)
- activation prerequisites can inspect accepted/pending state
- rows are persisted as Phase 4 handoff inputs (without running transcription yet)

## `layout_line_artifacts`

Prompt 43 adds canonical VLM-ready line artifacts anchored to layout IDs.

- `run_id`
- `page_id`
- `line_id`
- `region_id` (nullable)
- `line_crop_key`
- `region_crop_key` (nullable)
- `page_thumbnail_key`
- `context_window_json_key`
- `artifacts_sha256`

This table is the authoritative bridge from layout structure to downstream transcription inputs.

## `document_layout_projections`

Explicit document-level active layout pointer.

- `document_id`
- `project_id`
- `active_layout_run_id` (nullable)
- `active_input_preprocess_run_id` (nullable)
- `active_layout_snapshot_hash` (nullable)
- `downstream_transcription_state` (`NOT_STARTED | CURRENT | STALE`)
- `downstream_transcription_invalidated_at` (nullable)
- `downstream_transcription_invalidated_reason` (nullable)
- `updated_at`

Projection ownership rules:

- overview/triage/workspace defaults resolve through `active_layout_run_id`
- no implicit fallback to "latest run" for default activation semantics
- projection updates are explicit mutations, not inferred view-time behavior

## Lineage And Activation Rules

- reruns append a new `layout_runs` row
- reruns increment `attempt_number`
- reruns preserve `parent_run_id` lineage references
- reruns can set forward supersession via `superseded_by_run_id`
- historical run rows remain immutable

Activation rules in Prompt 44:

- only `SUCCEEDED` runs are activation candidates
- activation is blocked if `page_layout_results` are missing
- activation is blocked if any page result is not `SUCCEEDED`
- activation is blocked if any page is missing a persisted `layout_recall_checks` row
- activation is blocked if a `NEEDS_RESCUE` page has no `ACCEPTED` rescue candidate
- activation is blocked if any page still has `PENDING` rescue candidates
- activation updates:
  - `document_layout_projections.active_layout_run_id`
  - `document_layout_projections.active_input_preprocess_run_id`
  - `document_layout_projections.active_layout_snapshot_hash`
  - `document_layout_projections.downstream_transcription_state`
  - `document_layout_projections.downstream_transcription_invalidated_*`
- activation does not mutate historical run data

Downstream transcription state on activation:

- `NOT_STARTED` when no active transcription basis exists
- `CURRENT` when transcription projection already matches the activated layout run + snapshot
- `STALE` when activation supersedes the current transcription basis

Current block reason codes (from conflict payloads/logging):

- `LAYOUT_RUN_NOT_SUCCEEDED`
- `LAYOUT_RECALL_PAGE_RESULTS_MISSING`
- `LAYOUT_RECALL_STATUS_MISSING`
- `LAYOUT_RECALL_STATUS_UNRESOLVED`
- `LAYOUT_RECALL_CHECK_MISSING`
- `LAYOUT_RESCUE_ACCEPTANCE_MISSING`
- `LAYOUT_RESCUE_PENDING`

## Phase 4 Handoff Preparation

Prompt 44 does not perform rescue transcription, but it prepares canonical handoff data:

- ordinary line-based pages remain available via `layout_line_artifacts`
- rescue candidates persist explicit geometry and stable IDs suitable for `source_kind = RESCUE_CANDIDATE`/`PAGE_WINDOW` selection in Phase 4
- pages marked `NEEDS_MANUAL_REVIEW` remain explicit and are not treated as rescued
- token anchors remain deferred to later prompts

## Deferred Strictness

Prompt 39 intentionally does not implement:

- manual-edit and reading-order-edit run kinds
- automatic downstream transcription invalidation policies

Prompt 40 now covers canonical PAGE-XML and overlay cache contracts plus deterministic controlled-storage paths for those artefacts. The remaining stricter behaviors are owned by later layout and recall prompts.
